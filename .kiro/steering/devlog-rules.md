# Devlog Rules

## When to Update DEVLOG.md
After: new features, bug fixes, refactoring, architecture decisions, testing milestones, tool/config setup, integrations, performance/security work, planning/design completion.

## Entry Format
```markdown
## [YYYY-MM-DD HH:MM] - [CATEGORY]: [Brief Title]

### What Was Accomplished
- Specific work completed

### Technical Details
- Tech/frameworks used, DB changes, performance/security notes

### Decision Rationale
- Why this approach, trade-offs, constraints

### Challenges and Solutions
- Problems faced, how resolved, lessons learned

### Next Steps
- Follow-up tasks, future improvements
```

## Categories
FEATURE | BUGFIX | REFACTOR | CONFIG | DOCS | TESTING | RESEARCH | PLANNING | INTEGRATION | PERFORMANCE | SECURITY | DEPLOYMENT

## Entry Ordering
New entries go at TOP of file, immediately after "## Recent Activity" header. Newest first.

## Quality
- Enough detail for unfamiliar reader to understand
- Explain what, why, and how
- Include code snippets/commands when relevant
- Document failures too
